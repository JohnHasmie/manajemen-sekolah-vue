Admin · Kegiatan Kelas — Redesign mockup

Drops the legacy Guru → Mapel → Kegiatan drill-down (3 taps to reach any activity, no overview signal) and replaces it with a single recent-first hub modelled after the brand pattern admins already use on Jadwal / Rekap Nilai / Kehadiran. KPIs surface the school-wide activity volume at a glance; chip filters + type tabs let admin narrow without descending the tree; the detail screen layers Informasi + Daftar Siswa + Statistik like the teacher detail (Frame A of the teacher mockup) but with the admin's monitoring lens (per-class submission progress, no edit affordance).

Design intent & shared-component map

  • Why drop the drill-down. Today's Guru → Mapel → Kegiatan path costs 3 taps to reach any activity and gives no overview signal — admins can't tell at a glance how many kegiatan landed this week, which teachers are silent, where submission rates are dropping. Recent-first hub fixes both.
  • Frame A — Hub. Built on BrandPageLayout + BrandPageHeader + BrandKpiStrip (3-col: Total / Minggu Ini / Belum Submit). BrandFilterChipStrip under header (Kelas / Mapel / Guru / Status). Pill-style type tabs (Semua / Tugas / PR / Ulangan / Lainnya) using the same component pattern as the Buku Nilai header role-toggle. Each card = BrandListRow with two meta-pill rows + a thin dashed separator hosting the submit-progress bar.
  • Frame B — Detail. Reuses BrandPageHeader (title = activity name; subtitle = "Detail Kegiatan"), KPI strip (Submit / Belum / Rerata) with green / amber / brand-dark colors. Sections via the same card chrome the Mapel / RPP detail uses — icon disc + title + content. No edit button (admin is monitor-only). Print PDF + "⋯" remain in the header rail.
  • Frame C — Filter sheet. Vanilla AppFilterBottomSheet with five sections — Kelas + Mapel + Tipe + Periode as FilterChipGrid, Guru as _buildAutocompleteField (same pattern as the Jadwal Guru/Mapel chips added in Fix-1a, since the guru list can be hundreds long).
  • Frame D — Quick actions. AppBottomSheet with kebab-source: 5 rows mirroring the Mapel kebab pattern. The "Filter pakai Guru ini / Mapel ini" rows let admin pivot the hub to a teacher's or subject's slice in one tap — recovers the drill-down's intent without the cost.
  • Frame E — Empty states. Three variants on the same chrome (pristine / filter-empty / read-only AY) following the Jadwal TR.H pattern. Admin doesn't get an "Add" CTA because admin doesn't create kegiatan — the empty state is monitoring-flavored.
  • Frame F — Timeline (optional alt-view). Renders kegiatan as a vertical timeline grouped by day with a colored node per type, matching the teacher Frame B timeline. Same view-toggle component as teacher.
  • Backend. Hub fetches via GET /class-activities/admin-summary (new endpoint analogous to teacher-summary) returning paginated cards plus a kpi block ({ total, this_week, pending_submissions }). Reuse the existing per-activity show endpoint for Frame B; the submissions list is already fetched via the class_activity_submissions tables (Brand 3.11a/b).
  • Removed state. The legacy _showTeacherList / _showSubjectList / _selectedTeacherId / _selectedSubjectId flags + the three drill-down loaders (loadTeachers / loadSubjectsByTeacher / loadActivitiesBySubject) all become dead code once Frame A ships. Plan to soft-delete via the in-place rewrite pattern documented in CLAUDE.md.

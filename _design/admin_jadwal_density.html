Admin Jadwal — Density Layout for High-Conflict Slots

When a school has 20+ classes scheduled at the same lesson_hour (the bentrok screenshot showed up to 21 sessions at one slot), side-by-side lanes break down — each block would be ~7px wide. This mockup proposes an adaptive density layout: lanes for low-density (1–3), compact stack for medium (4–5), and an aggregator cluster card for high-density (6+). Tapping the cluster opens a sheet with the full list, scoped filters, and inline actions. Long-pressing it puts every session at that slot into bulk-select mode.

Why this layout

  • Density buckets, not infinite lanes. <= 3 sessions per slot: side-by-side lanes (existing). 4–5: compact 12px mini-cards stacked vertically with "+N lagi" tail. 6+: single aggregator card. This prevents the column from ever rendering blocks narrower than ~24px (the minimum readable width for "Mat 7A").
  • Cluster card carries the most-actionable signal first. The count badge ("21 sesi") and conflict status (red/cobalt) read at glance. Two preview lines hint at what's there — top mapel + tally — without trying to cram all 21 labels in.
  • Two interaction paths, no ambiguity. Tap → exploration (expansion sheet with search, filter tabs, inline actions). Long-press → bulk action (every session at that slot becomes a selection unit, then bulk-move / bulk-teacher / bulk-delete kicks in). Both are flows admins already learned in the existing redesign.
  • The expansion sheet is the actual table. Inside the sheet, density doesn't matter — it's a vertical scrollable list. Filter chips at the top (Semua / Bentrok / per-mapel) make the 21-row list searchable in two taps. Each row gets two icon-actions (✎ edit, ⇄ pindah) plus tap-to-open detail.
  • "+ Tambah di slot ini" closes the loop. If an admin opens a slot's expansion sheet and decides to add another session there, the day + lesson_hour are pre-filled. Saves 4 picker taps.

Implementation notes (for when we apply this)

  • Density bucket is computed per-cluster, not per-day. Reuses the sweep-line cluster detection already in _computeLanes. Add a renderMode field (lanes / stack / cluster) chosen by cluster size.
  • Cluster card is a single Positioned, full cell. Replaces all the per-block Positioneds for that cluster. Cheaper to render than 21 narrow rectangles.
  • Expansion sheet reuses AppBottomSheet + FilterChipGrid. No new shared widgets needed. The list rows are a thin variant of AdminScheduleRowCard (no time column — time is in the header).
  • Long-press → bulk select. Existing _toggleSelection already supports this; we just need to seed _selectedIds with every id in the cluster on long-press.
  • Drag-and-drop drops onto a cluster. Treat the cluster as a single drop target. Drop adds the dragged session to that cluster (and either succeeds or surfaces a 409 conflict toast — same as today).

Admin · Mata Pelajaran Detail · Flow redesign

Proposed redesign for the subject-detail screen and its supporting sheets. The current screen mixes a dialog-style add flow, a search-bar dropdown filter, and ad-hoc cards. This redesign aligns every surface with the shared brand contract (BrandPageHeader · BrandKpiStrip · BrandListRow · AppBottomSheet · BrandFilterChipStrip) and introduces the missing affordances: wali kelas assignment per class, multi-select add, sort + status filter, and a quick action sheet on assigned rows.

What's new vs. the current screen

  • Header chip strip — Status and Urutkan as two chips inside the gradient. Tapping either opens Frame B. Default sort is "Terdaftar dulu" so wali-kelas-needed rows surface immediately.
  • Inline wali kelas chip on each assigned row — indigo pill with a 2-letter avatar when set, amber "⚠ Wali belum diset" when missing. Tapping the chip opens Frame D directly (no detour through the kebab).
  • Subject meta card at the top of the body — Kode · Kategori · JP · KKM · Tingkat. Today the screen shows none of this; admin has to leave the screen to remember which mapel they're editing.
  • Multi-select add (Frame E) — replaces the one-at-a-time dialog. Tingkat tabs + live selection summary + batch Tambah n Kelas. Mirrors how Buku Nilai's bulk-attach flow works.
  • Quick-action sheet (Frame C) on the kebab — surfaces Ganti wali · Lihat jadwal · Buka buku nilai · Lepas. Today the only action is tap-to-toggle, which is destructive without confirmation.
  • Edit in brand chrome (Frame F) — same SubjectAddEditSheet underlying widget, wrapped in AppBottomSheet so the header pattern (kicker · title · icon · ✕) matches every other sheet on this screen. Adds a Zona berbahaya block for delete so it doesn't live as a separate trash icon in the AppBar.
  • BrandListRow rows — green leading icon + green status dot + Terdaftar label for assigned, neutral slate for unassigned, with a single "Detail" / "+ Tambah" trailing CTA so the action is discoverable without a long-press.

Open questions for you

  • Should the Wali kelas chip on the row open Frame D (current proposal) or open Frame C first? I picked direct because "fix the missing wali" is the most common reason admin opens that row.
  • Frame E currently shows 1 tab "Semua" plus per-tingkat tabs. Do you want to default to filtered by the subject's allowed tingkat (Matematika is VII–IX so only those tabs show), or show all and let admin add e.g. SD kelas if needed?
  • Frame D groups teachers as Saat ini / Tersedia / Sudah jadi wali. Do you want the "Sudah jadi wali" group hidden by default behind an "Tampilkan yang sudah jadi wali" toggle, or always shown like the mockup?
  • Frame F has a Zona berbahaya delete block. The current screen has no delete at all from inside the detail — should I keep delete here, or only on the list?
